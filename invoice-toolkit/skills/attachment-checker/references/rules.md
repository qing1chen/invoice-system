# 发票附件检查规则

> 本文件定义了各类别发票的附件检查规则。大模型读取后自动执行检查。
> 修改本文件即可调整检查逻辑，无需改动代码。

---

## 通用识别规则（v4 三大统一机制）

> **⚠ 本节描述所有类别共用的发票/附件识别与匹配机制，v4 起统一生效。**

### 1. 统一 OCR 名单区分发票与附件

**所有类别**（打车、出差、加班餐、打印、快递、材料等）统一使用 OCR 识别名单区分：
- **在 OCR 识别名单中** → 发票（已被百度 OCR 成功识别为发票的文件）
- **不在 OCR 识别名单中** → 附件（行程单、截图、情况说明等非发票文件）

不再依赖文件名关键词（如「发票」「滴滴」等）做发票/附件粗分。关键词仅作为
LLM 判断「发票是否属于当前类别」的辅助特征。

### 2. 附件统一截图+OCR 提取文字

所有附件（PDF、图片、doc/docx）统一通过"渲染为图片 → OCR 识别"管线提取文字内容，
提取结果传给 LLM 用于：
- **发票-附件精准配对**：如行程单中的出发地/目的地匹配对应发票
- **附件内容校验**：如支付截图金额是否与发票一致、情况说明中的人名/人数核对
- **语义匹配增强**：LLM 可基于附件实际内容（而非仅文件名）做匹配决策

### 3. 来源目录优先

附件匹配的**第一优先级来源**为 `data/课题组成员文件/<人名>/`（来源目录）。
在调用 LLM 决策前，系统会预先从来源目录收集该发票所属人员的所有候选附件，
与类别目录中的附件合并后一起传给 LLM 做最优匹配。

来源目录中的候选附件筛选条件：
- 来自指定人员的源文件目录（`source_root/<人名>/`）
- 不在 OCR 识别名单中（即非发票文件）
- 未被本次运行中其他发票匹配过

## 通用金额规则（所有类别均适用）

> **⚠ 本节规则独立生效，叠加在下面各类别规则之上。**

- **转账截图阈值**：**凡发票金额 > 1000 元，无论所属类别（打车、出差、加班餐、
  打印、快递、材料等），除该类别本身要求的附件外，必须额外提供「转账截图」。**
- **可接受的转账截图形式**：银行回单、付款截图、转账截图均可（满足其一即可）。
- **缺失标注**：若 > 1000 元发票没有转账截图，哪怕其他附件齐全也应标记为
  「缺少附件」，`missing_type` 写「转账截图」；若同时缺多类附件，用顿号连接，
  例如「行程单、转账截图」。
- **匹配策略**：转账截图按人名优先匹配，其次 LLM 语义匹配，最后回退到
  来源目录（`data/课题组成员文件/<人名>/`）搜索。

## 打车

- **必需附件**: 行程单
- **数据字段**: 价税合计, 销售方名称, 商品名称, 匹配简介, 姓名/公司
- **发票特征**: 发票, 滴滴, 出租, 网约车, uber, 曹操, T3, 首汽, 美团打车, 高德打车, 花小猪
- **附件特征**: 行程, 行程单, 行程明细, 行程记录, 乘车记录, 打车记录, 出行记录, ride
- **检查规则**: 每张打车发票必须有对应的行程单。附件来源优先从 `data/课题组成员文件/<人名>/` 收集，与类别目录附件合并后由 LLM 统一配对。配对依据包括人名、文件名语义以及**附件 OCR 提取的文字内容**（如行程单中的出发地、目的地、金额等）。**附件内容校验**：LLM 可基于行程单 OCR 内容核对行程信息是否与发票一致。**此外若金额 > 1000 元，还需额外提供转账截图，见「通用金额规则」节。**

- **数据字段**: 价税合计, 销售方名称, 商品名称, 匹配简介, 姓名/公司
- **发票特征**: 火车, 高铁, 动车, 飞机, 机票, 航空, 客票, 车票, 12306

- **检查规则**: 出差发票需要对应的行程单。附件来源优先从 `data/课题组成员文件/<人名>/` 收集，与类别目录附件合并后由 LLM 基于人名、文件名语义及**附件 OCR 内容**统一配对。**此外若金额 > 1000 元，还需额外提供转账截图，见「通用金额规则」节。**

## 加班餐

- **必需附件**: 情况说明
- **数据字段**: 价税合计, 销售方名称, 商品名称, 姓名/公司
- **发票特征**: （无特定关键词——通过 OCR 识别名单统一区分，与其他类别一致）
- **附件特征**: 情况说明, 说明, 加班说明, 加班餐说明
- **自动生成**: 是
- **检查规则**:
  1. **发票/附件区分**: 与所有类别一致，通过 OCR 识别名单统一区分——在 OCR 名单中 = 发票，不在 = 附件。
  2. **校验情况说明**: 如果发票已有对应的情况说明（.doc/.docx），通过 **OCR 提取文字内容**后校验以下内容：
     - **人数校验**: 人数 × 30 ≥ 发票金额（即每人最高标准 30 元，人数不少于 金额 ÷ 30 向上取整）
     - **人名校验**: 所有人名必须在课题组成员名单（`NAME_LIST`）中
     - **人名顺序**: 报销人应排在人名列表首位
     - **金额校验**: 情况说明中记载的金额应与发票金额一致
  3. **自动修复**: 人名不在名单、人数不足、金额不匹配时，自动修复情况说明文档（修改前备份到 `cache/attachment_backup/`）：
     - 替换无效人名为名单中的有效人名
     - 补充人数至满足 人数 × 30 ≥ 金额
     - 确保报销人排在首位
     - 修正金额
  4. **自动生成**: 如果发票没有对应的情况说明，自动生成 .docx 文件（山东大学科研业务专项经费使用说明表格式），内容包含加班事由、就餐地点、就餐费用、参加人名。
  5. **自动合并**: 生成/修复完成后，同一人的多份情况说明自动合并（最多 6 个一组），合并后删除原始散件。
  6. **转账截图（通用规则）**: 若加班餐发票金额 > 1000 元，除情况说明外还需额外提供转账截图，见顶部「通用金额规则」节。

## 材料

- **必需附件**: 转账截图 / 验收单（条件性 — 转账截图部分见「通用金额规则」）
- **数据字段**: 价税合计, 商品单价, 销售方名称, 商品名称, 匹配简介, 姓名/公司
- **发票特征**: 发票, 增值税
- **附件特征**: 截图, 转账, 付款, 验收, 验收单, 支付, 付款截图, 转账截图, 银行回单
- **检查规则（分级）**:
  1. **单价 > 1000 元 或 金额 > 20000 元**（材料类额外要求）: 除转账截图
     外，还需要**验收单**。
  2. 附件来源优先从 `data/课题组成员文件/<人名>/` 收集，与类别目录附件合并后
     由 LLM 基于人名和**附件 OCR 内容**统一匹配（分别按「转账截图」和「验收单」
     两类独立搜索）。
  3. **附件内容校验**（v4 新增）：LLM 基于转账截图 OCR 提取的文字内容，
     校验截图中的金额是否与发票金额一致、收款方是否与发票销售方一致等。
  4. 两类附件独立检查：可能缺一个（只缺转账截图或只缺验收单），也可能两
     个都缺。

## 通用匹配策略（v4 来源优先 + 内容驱动）

以上所有类别的附件匹配遵循统一的**来源优先 + 内容驱动**策略：

1. **来源目录优先**（v4 新增）: 在 LLM 决策前，预先从 `data/课题组成员文件/<人名>/` 
   收集该发票所属人员的所有候选附件，与类别目录附件合并
2. **OCR 内容提取**（v4 新增）: 对所有候选附件（PDF、图片、doc/docx）统一通过
   截图+OCR 提取文字内容，传给 LLM 用于精准配对和内容校验
3. **LLM 统一决策**: LLM 基于人名、文件名语义、**附件 OCR 文字内容**三维信息
   做最优配对，同时完成内容校验（如金额核对、人名核对等）
4. **最终兜底**: 若 LLM 判定缺少附件，再次搜索来源目录中尚未使用的候选文件
   （含 OCR 内容），做最后一次匹配尝试

## 异常标记格式

检查结果的异常标记遵循以下格式：

- 缺少附件: `[类别]缺少XXX`，例如 `[打车]缺少行程单`
- 校验不通过: `[类别]附件校验不通过: 详情`，例如 `[加班餐]附件校验不通过: 人名不在名单: 张X`
- 多条异常用分号 `;` 分隔追加

## 文件名规范

检查附件时也可选检查发票文件名是否符合「报销人+金额+用途」格式：
- 格式：`人名+金额+用途.pdf`，用 `+` 连接
- 金额保留小数，不含「元」字
- 用途尽量保留原有信息，保留角色标识（清单、截图、行程单等）
- 不符合规范的文件由 LLM 建议新文件名
