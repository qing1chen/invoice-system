# LLM 多步骤流程管理模式

> 本文档解释"LLM 编排多步骤工作流"的架构模式，供理解和扩展此 Skill 参考。

---

## 核心架构：LLM 是指挥官，脚本是工具

```
┌──────────────────────────────────────────────┐
│          宿主 LLM（Claude）                    │
│                                              │
│  ┌─ 读 SKILL.md ─ 读 rules.md ─ 做决策 ────┐ │
│  │                                          │ │
│  │   步骤1 → 步骤2 → ... → 步骤N           │ │
│  │     ↓        ↓              ↓            │ │
│  │  bash调用  bash调用       bash调用        │ │
│  └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
         ↕            ↕            ↕
┌─────────────────────────────────────────────┐
│        scripts/tools.py (CLI 入口)           │
│  python -m scripts.tools <name> '<json>'     │
├─────────────┬──────────┬──────────┬─────────┤
│  文件IO     │ 数据库   │  OCR     │ docx    │
│  collect    │ lookup   │  百度API │ 生成    │
│  copy       │ upsert   │  提取    │ 修复    │
└─────────────┴──────────┴──────────┴─────────┘
```

**关键点**：
- **不存在内部 Agent 循环** —— 宿主 Claude 直接读 SKILL.md，自己按步骤
  通过 bash 调用 `scripts/tools.py`
- `tools.py` 是纯工具集，通过 CLI 入口接收 JSON 参数、返回 JSON 结果
- 所有业务决策（发票/附件区分、匹配、校验）由宿主 Claude 完成

| 维度 | 传统方案（内嵌Agent循环） | 标准 Skill（宿主直接编排） |
|------|--------------------------|--------------------------|
| 流程控制 | Python `_run_agent` 循环 | **宿主 Claude 自然语言推进** |
| 工具调用 | function-calling + 路由 | **bash 直接调用 CLI** |
| 业务判断 | 内嵌 LLM prompt chain | **宿主 Claude 直接决策** |
| 规则变更 | 改 rules.md | **改 rules.md**（一致） |
| 状态管理 | Agent 对话上下文 | **宿主 Claude 对话上下文** |
| 额外依赖 | LLMClient + tool_definitions | **无额外依赖** |

---

## 过程管理的四个核心机制

### 1. 检查点（Checkpoint）

在长流程中，LLM 不能一口气跑完所有步骤再回话——用户需要看到进度，
而且上下文窗口有限。解决方案是**检查点**：

```
步骤 0: 加载规则 → ✓ 汇报"已加载 6 个类别的规则"
步骤 1.x (打车): ... → ✓ 汇报"打车：8张发票，6齐全，2异常"
步骤 1.x (出差): ... → ✓ 汇报"出差：5张发票，全部齐全"
步骤 1.x (加班餐): ... → ✓ 汇报"加班餐：12张发票，4需生成"
...
步骤 4: 汇报最终结果
```

**检查点的设计原则**：
- 每完成一个"用户可理解的逻辑单元"时设置
- 检查点消息要简洁（一行摘要），不要倾倒原始数据
- 如果某步骤耗时长（如 OCR 提取），可以在开始和结束各报一次

### 2. 上下文状态管理

LLM 没有持久化存储，但它有**对话上下文窗口**。
状态管理策略：

**方式 A：累积结果列表**（推荐用于本 Skill）
```
// LLM 在上下文中维护的数据结构
all_results = []

// 处理打车类别后
all_results.append({ 发票:"张三+50.pdf", 状态:"附件齐全", ... })
all_results.append({ 发票:"李四+1200.pdf", 状态:"缺少附件", ... })

// 处理出差类别后
all_results.append({ 发票:"王五+300.pdf", 状态:"附件齐全", ... })

// 最后一次性传给 save_attachment_report
```

**方式 B：写入中间文件**（适用于超长流程，避免上下文溢出）
```
// 每处理完一个类别，就把结果写入临时文件
// 最后从文件读取全部结果
```

**方式 C：分批写入数据库**（适用于数据量极大的情况）
```
// 每处理完一个类别，立即调用 save_attachment_report 写入
```

### 3. 循环与迭代

LLM 没有 `for` 循环语句，但可以通过**叙述式迭代**实现：

```
我需要处理 6 个类别：打车、出差、加班餐、打印、快递、材料。

先处理【打车】：
  1. 收集文件...（bash 调用工具）
  2. 区分发票/附件...
  3. ...
  结果：打车处理完毕，8张发票。

接下来处理【出差】：
  1. 收集文件...（bash 调用工具）
  ...
```

**关键技巧**：
- 在开始前列出完整的迭代清单，这样不会遗漏
- 每轮迭代开始时明确声明"正在处理X"
- 跟踪进度："已完成 3/6 个类别"

### 4. 分支与决策树

LLM 的优势在于**灵活的分支决策**。不需要穷举 if-else，而是基于规则理解做判断：

```
对于这张发票（张三+1200+出租车.pdf）：
  - 金额 1200 > 1000，需要转账截图（通用规则）
  - 是出租车发票，需要行程单（打车规则）
  - 在候选附件中找到"张三行程单.jpg"，OCR 内容包含出租车行程 ✓
  - 但没找到转账截图 ✗
  → 判定：缺少附件，缺少类型=转账截图
```

这种推理在 Python 中需要大量嵌套条件和特殊处理，LLM 天然擅长。

## 文件结构

```
attachment-checker/
├── SKILL.md                    ← 宿主 Claude 读取的编排指令
├── references/
│   ├── rules.md                ← 业务规则，随时可改
│   ├── tools.md                ← 工具函数参考文档
│   └── process-management.md   ← 本文件，架构说明
└── scripts/
    └── tools.py                ← 纯工具函数 + CLI 入口
```

**设计原则**：
- `SKILL.md` 告诉宿主 Claude 做什么、怎么做
- `rules.md` 告诉宿主 Claude 检查规则是什么
- `tools.md` 告诉宿主 Claude 有哪些工具可用
- `tools.py` 是被 bash 调用的执行层，不含任何业务逻辑

## 扩展新步骤

要给工作流添加新步骤：

1. **在 SKILL.md 中描述新步骤**：在流程骨架中插入位置，写清楚 LLM 要做什么决策
2. **如果需要新工具**：在 `scripts/tools.py` 中添加函数及 CLI 路由，在 `references/tools.md` 中添加文档
3. **如果需要新规则**：在 `references/rules.md` 中添加规则段

**示例：新增"附件内容深度校验"步骤**

```markdown
### 步骤 1.5.4 附件内容深度校验（新增）

对已匹配的附件，进一步校验内容与发票的一致性：

**你的决策**：
- 转账截图：OCR 文字中的金额是否与发票金额一致？收款方是否匹配？
- 行程单：行程日期是否在发票开票日期前后？出发地/目的地是否合理？
- 明细/清单：项目是否与发票商品名称对应？

如果内容不一致 → 更新状态为"附件校验不通过"，记录具体不一致项。
```

不需要改任何 Python 代码——LLM 会基于已有的 OCR 文字内容自行判断。
